Why proof repair is the right pressure point

The proof-repair lane is useful because it is repeated, concrete, source-backed, and risky enough to need controls. The latest detector found five failed approved-send artifacts. That means the system can prove a worker intended or attempted a communication, but no sent Message-ID was produced. This is exactly where loop engineering helps: the loop should not guess, should not auto-send, and should not mark work complete. It should build a packet, show the evidence, require approval, execute through the correct lane path, and verify the live proof afterward.

Operating cadence

Daily:

- Run loop status.
- If next_action is monitor_mode, record readback and stop.
- If next_action is review_proof_repair_packets_for_approval, review packets and ask for explicit approval where needed.
- If drift exists, repair drift before returning to monitor mode.
- If a monitor is degraded, classify it before launching more workers.

Weekly:

- Review recursive proposal outcomes: kept, rejected, blocked, superseded.
- Retire proof-backed stale worktrees.
- Refresh repo/live hygiene inventory.
- Pick one repeated manual operation to convert into a verifier-backed loop.

Before any mutation:

- Confirm approval contract.
- Confirm narrow mutable surface.
- Run preflight verifier.
- Apply the smallest diff possible.
- Run post-change verifier and live/read-model proof.
- Record the result in the durable task spine.
