Use the saved critical package file as the source of truth after hydration. If hydration remains blocked, use the project note package groups only as a checklist and stop before image build.

Minimum package groups that must be present in the generated custom image manifest:

- Router access: `dropbear`, LuCI, `luci-ssl`, `uhttpd`, `uhttpd-mod-ubus`, `rpcd`/LuCI dependencies.

## Evaluation Workflow

1. Hydrate only the non-secret local source artifacts needed for evaluation: critical package list, installed package list, official `sha256sums`, official `profiles.json`, and existing official image metadata.
2. Re-run local hashes for the official files and compare to the official online hashes. Stop if any local file hashes as empty or mismatched.
3. Build a custom package list from the hydrated critical package file and project-note package groups. Keep it as a text input; do not copy secret config into the builder.
4. Choose builder path for the next separately approved evaluation step:
   - Preferred simple path: official Firmware Selector/custom image flow for OpenWrt `25.12.2`, target `mvebu/cortexa9`, profile `linksys_wrt3200acm`, package list from the critical package input.
   - Local reproducible path: download and hash-verify `openwrt-imagebuilder-25.12.2-mvebu-cortexa9.Linux-x86_64.tar.zst`, then build locally using profile `linksys_wrt3200acm` and the vetted package list.
5. Preserve generated build artifacts in a new local folder under `.private/router/openwrt-upgrade-20260414-1000/custom-image-evaluation/`:
   - package input list
   - builder method and command/API request, with no secrets
   - generated manifest
   - generated image filename, size, and SHA256
   - comparison notes against required package groups
6. Accept the custom image for validation-only staging only if the generated manifest contains the required IKEv2/WireGuard/LuCI/firewall/DNS package groups and target/profile metadata still matches WRT3200ACM.
7. Stop for Robert approval before any router-side custom-image staging or `sysupgrade -T` for the custom image.

- Downloading the ImageBuilder tarball or building/requesting a custom image is the next evaluation step, but it is not part of this checklist-only pass unless Robert separately approves it.
- Copying a custom image to the router requires a separate validation-only approval for the exact file and hash.

- Flashing any image requires a later separate explicit approval.
- Any reboot, firewall reload, VPN reload, StrongSwan/WireGuard restart, or live package install remains outside this approved scope.

## Decision Rule

Proceed toward a custom image only if it improves preservation confidence over the plain official image. If the package manifest cannot be verified cleanly, stop and keep the official image as a validated fallback only for a future maintenance window with local/LAN recovery ready.
