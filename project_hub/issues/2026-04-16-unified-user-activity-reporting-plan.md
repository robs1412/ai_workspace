## Scope

Define the planning boundary for a unified user activity reporting layer across KOVAL internal systems. This slice records source systems, read-only data surfaces, privacy/security gates, cadence, first implementation slice, and workspace ownership.

This pass is documentation only. It does not inspect credentials, connect to production databases, fetch production rows, send email, mutate Google Workspace/Admin/Gmail settings, implement code, deploy, or start background polling.

## Reporting Goal

Create a management-facing view of internal system adoption and operational activity by user, team, workspace, and time period. The report should answer:

- Which users are active in the internal systems they need for their role?
- Which operational workflows are being used regularly, including tasks, shifts, check-ins, events, reports, imports, and automation?
- Which systems need follow-up training, access correction, or workflow cleanup?
- Which metrics can be trusted as read-only operational facts, and which are only advisory overlays?

The first deliverable should be a source inventory and metric contract. A dashboard or recurring email is a later phase.

## Read-Only Data Surfaces

Allowed planning and first-slice surfaces:

- Local docs, TODOs, handoffs, project-hub records, and code-defined table/query references.
- Read-only schema inspection in target repos after Code and Git Manager preflight when code work begins.
- Existing app-level read-only reports where already exposed to approved users, such as OPS task stats or usage-stats screens.
- Read-only SQL views or SELECT-only queries only after workspace owner approval and with production credentials handled privately outside chat/docs.
- Export files generated by system owners for this purpose, stored outside the repo if they contain sensitive or production-derived data.
- Aggregated log metadata where paths are approved and contents do not include secrets, message bodies, or private notes.
- Workspaceboard metadata and concise summaries, not raw transcripts when they may include private data.

Disallowed in this planning slice:

- Credential lookup, `.env` printing, OAuth token inspection, private key access, or app password handling.
- Production DB connections or row dumps.
- Gmail mailbox reads, email body extraction, or Google Admin report access.
- Any live writes, task/email/notification sends, deploys, service restarts, or runtime/code changes.
- Background polling daemons.

## Privacy And Security Gates

- Security Guard must review any future slice that touches Google Workspace Admin, Gmail, mailbox metadata, OAuth scopes, service accounts, `.205`, Keychain, system permissions, external APIs, or cross-machine credentials.
- Human approval is required before any production data pull, Google Admin export, mailbox metadata access, new read-only service account, OAuth app/scope change, live scheduled report, or staff-facing report distribution.
- Default report granularity is aggregate by user/team/time period. Raw activity rows are for verification only and should not be broadly stored.
- Email content, private notes, credentials, tokens, reset links, OAuth cache files, private mailbox contents, and sensitive personnel/HR details must not enter the reporting dataset.
- Where user-level reporting could be interpreted as performance monitoring, the report must state its operational purpose: adoption, access, training, and workflow health.
- Reports must separate facts from interpretations. For example, "no login in 30 days" is a fact; "user is not doing the job" is not an allowed inference.
- Any dashboard or recurring email must have an explicit audience, owner, redaction level, duplicate protection, and opt-out/disable procedure before launch.
- Store implementation notes in workspace/project docs; store any sensitive exports outside git with manifests/checksums only when needed.

## Cadence

Recommended cadence after the source inventory is approved:

- Daily: internal automation health and obvious failed-run counters for workers/scheduled jobs, only to the owning operations channel or Task Manager.
- Weekly: adoption/access review for system owners, focusing on missing access, stale users, failed imports, and training follow-ups.
- Monthly: management-facing user activity overview with task stats, shift/check-in stats, report usage, import cadence, and notable automation activity.
- Quarterly: source inventory and metric-definition review, including retired systems, new modules, role changes, and privacy/security re-approval for overlays.

Do not enable recurring sends in the planning slice. A later send/reporting slice must define recipient list, copy, scheduler, duplicate protection, and approval gate.

## First Slice

First implementation slice, when approved:

1. Keep execution in `ws ai` as a planning/coordinator task, then open source-specific workers only for source inventory.
2. For each source workspace, inspect only docs/code/schema references needed to list candidate read-only metrics. Do not connect to production data.
3. Produce a metric contract with metric name, owner, source table/report/view candidate, join key, time grain, privacy level, verification method, and approval gate.
4. Pick one low-risk MVP report: monthly aggregate task/check-in/report-usage counts by active user, with no email/Gmail/Admin overlay.
5. Review the contract with Security Guard before any live data access or recurring report design.

Suggested MVP sources:

- `ws login`: canonical active user and role/access map.
- `ws ops`: aggregate task stats by owner/assignee/status/month.
- `ws portal`: aggregate check-in/shift participation by user/month.
- `ws sales`: aggregate salesreport login/report usage where a read-only source exists.

Explicitly defer:

- Gmail/Google Admin/Gemini overlays.
- Email-derived Frank/Avignon metrics beyond aggregate task counts from their local logs.
- Finance/BID interpretation beyond usage/refresh metadata.
- Dashboard UI, scheduled email, or production read replicas.
