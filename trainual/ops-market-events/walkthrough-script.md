# OPS Market Events Walkthrough Script

Status: planning-only draft
Recording style: manual-paced, pointer visible, no overlays unless Robert separately requests them

## Opening

"This walkthrough covers how to use Market Events in OPS for day-to-day event work. We will look at where to find the event list, how to read an event record, what to check before an event, and what to update afterward."

"This recording uses safe demo data. Do not copy demo names, dates, or account details into live work."

## Scene 1: Start From OPS

Action:

- Begin after the user is already logged in.
- Keep the login, password, 2FA, and account setup screens out of the recording.
- Show the normal OPS starting point.

Narration:

"Start from OPS after you are logged in. If you cannot access OPS or the Market Events area, stop and ask for access help. Do not use another person's login."

## Scene 2: Open Market Events

Action:

- Move the pointer to the Market Events navigation entry.
- Open the Market Events list or calendar view.
- Pause briefly after the page loads.

Narration:

"Open Market Events from the OPS navigation. This is the working area for tastings, samplings, and related market activity."

## Scene 3: Orient To The List

Action:

- Point to the main date, market, account/location, owner, status, and filter areas.
- If filters are visible, demonstrate checking them without changing live data.

Narration:

"Before opening a record, check the filters and date range. A common mistake is thinking an event is missing when the list is filtered to another market, owner, or date."

## Scene 4: Open A Demo Event

Action:

- Open a clearly labeled demo or seeded event.
- Pause after the detail page opens.

Narration:

"Open the event record you plan to work on. In training, use a clearly marked demo event. In live work, make sure the date, account, and owner match the event you meant to open."

## Scene 5: Read The Event Details

Action:

- Point to event name, date/time, location/account, owner, market, status, notes, and linked shift or staffing sections if present.
- Do not expose private notes or personal contact information.

Narration:

"The key details are the event name, date and time, account or location, market, owner, status, and notes. If staffing or shifts are linked, confirm those details before the event."

"Some fields are informational. Only update fields that are part of your role or that the event owner has asked you to update."

## Scene 6: Pre-Event Check

Action:

- Show the sections a user should review before the event.
- If there is a related account/contact/activity area, point to it without creating live records.

Narration:

"Before the event, confirm the time, location, staffing, and any account notes. If related account or contact information is needed, search before creating anything new so duplicates do not get added."

## Scene 7: After-Event Update

Action:

- Show where outcome notes or completion status would be entered using demo data only.
- If the actual recording cannot safely save a record, stop before submit and explain in the notes that no live save was performed.

Narration:

"After the event, record the outcome in the expected notes or activity area. Keep notes factual and work-related. Do not put passwords, private personal details, or unrelated internal comments into the event record."

"Only change status when the event is really complete, cancelled, or otherwise confirmed."

## Scene 8: Wrap Up

Action:

- Return to the list or leave the event detail page visible with no private data.

Narration:

"Use Market Events as the source for event work, check filters before assuming a record is missing, search before creating related records, and ask the event owner when ownership or instructions are unclear."

## Recording Notes To Capture Separately

- Date recorded.
- OPS environment used: demo, staging, seeded local, or live read-only.
- Whether any save action was performed. Planning default: no.
- Demo records used.
- Any screens intentionally skipped for security or privacy.
- Known differences between the recording and live production behavior.
